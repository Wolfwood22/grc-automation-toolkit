Project Walkthrough
Serverless IAM Access Review System

An automated access review pipeline built on AWS that turns IAM control checks into auditor-ready output on a weekly cadence, with no human in the loop.

python aws lambda cloudformation iam · security hub · eventbridge · s3
01 //

Problem Statement

Manual access reviews are one of the most consistently failed controls in SOC 2 and ISO 27001 audits — not because teams don't understand the requirement, but because the process is time-consuming, inconsistent, and easy to defer under audit pressure. When it does happen, the output is often a spreadsheet someone assembled the night before the assessment — data an auditor has to accept without any way to verify it.

The goal was a system that runs without human intervention, checks the controls that actually matter, and produces a timestamped, self-contained report that lands in S3 automatically. Something an auditor can open and act on, not something someone had to put together under deadline pressure.

02 //

Architecture Decision

The serverless choice wasn't a default — it was a GRC-informed infrastructure decision. Every component was chosen because it reduces audit surface, not just operational overhead.

// no server
Zero Patch Surface
No EC2 instance to patch, harden, or monitor. Lambda eliminates an entire class of infrastructure findings before an auditor ever looks.
// eventbridge
Auditable Cadence
Weekly trigger means the review runs on schedule, every time, with a CloudWatch execution log. The cadence itself becomes audit evidence.
// s3 output
Immutable Evidence Trail
Reports land in S3 as access-review-YYYYMMDD.html — timestamped, versioned, and retrievable by auditors without any system access.
// system architecture
03 //

What the System Actually Checks

Each finding source maps to a specific compliance risk — not just a configuration gap. The system checks the things auditors actually test for.

iam_findings.py
IAM Identity Hygiene
Flags stale accounts with no activity in 90+ days, admin users missing MFA, and IAM users without required manager tags — the three access control failures that appear most consistently in SOC 2 and ISO 27001 audit findings.
Maps to: SOC 2 CC6.1 · ISO 27001 A.9 · NIST 800-53 AC-2
access_analyzer.py
External Resource Exposure
IAM Access Analyzer surfaces cross-account trust relationships, publicly accessible S3 buckets, and other resource policies that grant access outside the account boundary — the kind of finding that causes critical findings in FedRAMP and PCI assessments.
Maps to: NIST 800-53 AC-3 · PCI-DSS 7.2 · CMMC AC.L2-3.1.3
security_hub.py
Active IAM Control Failures (IAM.1–IAM.9)
Pulls active FAILED findings for all nine AWS Security Hub IAM controls — root MFA, password policy, unused credentials, access key rotation — pre-mapped to NIST 800-53 and PCI-DSS so findings are immediately actionable for auditors without translation.
Maps to: NIST 800-53 IA-2 · IA-5 · PCI-DSS 8.x · CMMC IA.L2
04 //

The Output: HTML Report with Executive Narrative

The report module generates a self-contained HTML file — no external dependencies, no server required to view it. An auditor can open it directly from S3 or download it and view it offline. This is a deliberate design choice: the report is the evidence artifact, not a dashboard that requires system access to interpret.

The narrative module generates an AI-written executive summary via Bedrock — translating raw IAM findings into business-readable risk language. This is the bridge between technical output and GRC-consumable evidence: the kind of summary a CISO or auditor can read without parsing JSON.

05 //

Compliance Frameworks Addressed

The report structure maps findings directly to control references across six frameworks — an auditor can use the output as evidence without any additional translation work.

SOC 2
CC6.1, CC6.2, CC6.3 — logical and physical access controls, user provisioning and deprovisioning
ISO 27001
A.9 Access Control — user access management, user responsibilities, system and application access
NIST 800-53
AC-2, AC-3, IA-2, IA-5 — account management, access enforcement, identification and authentication
PCI-DSS 4.0
Requirements 7 and 8 — restrict access to cardholder data, identify users and authenticate access
CMMC Level 2
AC.L2-3.1.1 through 3.1.3, IA.L2-3.5.3 — access control and multi-factor authentication practices
CMMC Level 3
Extended access control and audit requirements mapping to NIST 800-172 enhanced security requirements
06 //

Demo Mode

The deployed portfolio version runs with DEMO_MODE=true — the system generates realistic synthetic findings rather than pulling live IAM data from a real AWS account. This means you can run and share the system without exposing production credentials or real access review results.

// note
Choosing to demo safely rather than expose live IAM data is itself a security judgment call — the kind that GRC-mature engineers make. It also means anyone can deploy the stack and see real output immediately without needing a pre-configured AWS environment.
// deploy via cloudformation
# Package and deploy the stack
aws cloudformation package \
  --template-file template.yaml \
  --s3-bucket your-deployment-bucket \
  --output-template-file packaged.yaml

aws cloudformation deploy \
  --template-file packaged.yaml \
  --stack-name iam-access-review \
  --parameter-overrides DemoMode=true \
  --capabilities CAPABILITY_IAM

# Stack provisions Lambda, EventBridge rule, S3 bucket, and IAM role
